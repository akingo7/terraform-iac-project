You are an expert Cloud Architect and Terraform specialist. I need your help to refactor my monolithic, flat AWS 3-tier infrastructure codebase into a professional, scalable, and modularized structure inside my working directory root.

I want to leverage VS Code Copilot to create and coordinate this refactoring. Please guide me through generating each file, ensuring we adhere to DRY (Don't Repeat Yourself) principles, Terraform best practices, and clean input/output mapping.

1. Key Guidelines & Architecture Rules
   No Functional Configuration Changes (Strict Constraint): During the refactoring process, do NOT alter any of the functional settings of the infrastructure. All subnet CIDR blocks, security group ports, resource naming conventions, KMS keys, database parameters, and target group path settings must remain exactly as they were in the flat, monolithic codebase. Only restructure the code logically into child modules and establish clean output/input dependencies.

S3 Remote Backend & Locking: We are using an S3 bucket as our remote backend with native S3 state-locking (note: we are explicitly omitting DynamoDB for locking as S3 native object lock is our standard here).

No Hardcoding: All values must be mapped to variables in their respective variables.tf files and overridden dynamically in terraform.tfvars at the root.

Refactoring Tools:

Security Groups: Use dynamic blocks for ingress/egress rules to keep the security module highly maintainable.

Compute (AMIs): Use map and lookup() functions to dynamically fetch CentOS AMI IDs based on the selected AWS Region (var.region).

Conditionals: Use Ternary Operators (condition ? true : false) where resource-level decision making is required.

1. Step-by-Step Generation Request

Please generate the Terraform configurations block-by-block, starting with the root files, followed by individual child modules. For each child module, provide the main.tf, variables.tf, and outputs.tf configurations:

Step 3.1: Root Core Configuration Files

providers.tf: Standard AWS provider with region variables.

variables.tf & terraform.tfvars: Root-level variable definitions and their respective default/override values.

main.tf (Root): Orchestrate calls to our modules, dynamically feeding outputs from earlier modules into subsequent modules (e.g., module.network.vpc_id passed into module.security.vpc_id).

Step 3.2: Module Definition Breakdowns

network Module: Handles VPC, 6 Subnets (2 Public, 2 Private Web, 2 Private Data), Internet Gateway, NAT Gateways, Elastic IPs, and Route Table associations.

security Module: Contains Security Groups for ALB, Bastion, Nginx, Webservers, and Data Layer using dynamic ingress rules.

ALB Module: Deploys External and Internal Application Load Balancers, Target Groups, Listeners, and maps ACM SSL Certificates on port 443.

EFS Module: Deploys EFS, Mount Targets in private data subnets, and Access Points.

RDS Module: Deploys S3-ready Subnet Groups, KMS-encrypted MySQL 8.0 instances, and Multi-AZ replication properties.

autoscaling Module: Launch templates and Auto Scaling Groups (Desired: 2, Min: 1, Max: 4) for Nginx, WordPress, and Tooling.

compute Module: Dedicated Bastion instances and static management nodes.
